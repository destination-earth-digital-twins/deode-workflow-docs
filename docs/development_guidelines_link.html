Development guidelinesïƒ

Describes best practices and guidelines for development in the Deode-Prototype repository.

Best practicesïƒ

  • Authors of PRs should not merge themself - Instead request a reviewer and an assignee. Assignee is responsible for merging.

  • Assignee and reviewer may not be the same person.

  • Authors of PRs are responsible for ensuring that the PR is up to date with the base branch before merging.

  • Assignees should not merge PRs that have failing tests.

  • Commits to master and develop branches should only be via PRs.

  • We rely on the good sense of both author and reviewer.

Checklist for authorsïƒ

  • âœ”ï¸ Make sure your local environment is correctly initialised as described in the README file.

  • âœ”ï¸ Use forks for your changes

  • âœ”ï¸ If not up-to-date, update your fork with the changes from the target branch (use pull with --rebase option if possible).

  • âœ”ï¸ Describe what the PR contains.

  • âœ”ï¸ Ensure your PR does not contain seperate features.

Checklist for reviewersïƒ

Each PR comes with its own improvements and flaws. The reviewer should check the following:

  • âœ”ï¸ Is the code readable?

  • âœ”ï¸ Is the code well tested? (Look at the coverage report)

  • âœ”ï¸ Is the code documented?

  • âœ”ï¸ Is the code easy to maintain?

Checklist for assigneesïƒ

  • âœ”ï¸ Is the PR up to date with the base branch?

  • âœ”ï¸ Are the tests passing?

  • âœ”ï¸ Is the PR ready to be merged?

  • âœ”ï¸ Squash commits and merge the PR.

Local testingïƒ

No-one likes to wait for the CI to run tests. It is therefore recommended to run tests locally before pushing to the remote repository, and before creating a PR, but no one will force you to do this: how you work locally is entirely up to you.

For convenience, however, we have added a few commands you can use to check that the code is linted, the tests pass, etc. Some of these are exemplified in the next subsections. Please run inside of your poetry shell:

poetry devtools -h

for more information.

Run linters and exit with an error if non-linted code is detectedïƒ

poetry devtools lint

Run linters and attempt to fix eventually encountered errorsïƒ

poetry devtools lint --fix

This will stop with an error if the encountered issues cannot be fixed.

Runthe typical checks for things you need to fix prior to a pushïƒ

poetry devtools pre-push-checks

Run testsïƒ

pytest

or

poetry devtools pytest

Generate and view the documentation to be published to our docpagesïƒ

poetry devtools doc clean
poetry devtools doc build
poetry devtools doc view

or, combining them all:

poetry devtools doc

Testing on Atosïƒ

The testing procedure above does not test the full functionality together with the IAL code. While waiting for a automated CI/CD system to be in place a few manual steps are required on atos to check the functionality.

  • âœ”ï¸ Run the default config file, using CY48t3, under ecflow following the instructions in the README file.

  • âœ”ï¸ Repeat the same using the config file for CY46h1 and CY48t3_Alaro, i.e. deode/data/config_files/config_[CY46h1|CY48t3_Alaro].toml.

  • âœ”ï¸ Test the nesting procedure for both CY46h1 and CY48t3 using the config files deode/data/config_files/config_[CY46h1|CY48t3]_target.toml. Both of the runs will use data from the first two tests.

  • âœ”ï¸ Finally test the stand alone task for the forecast following the instructions in the README file.

Branchesïƒ

As of now, the repository has two main branches:

  • master - This is the main branch. It is the branch that is deployed to production.

  • develop - This is the branch that is deployed to staging. As the project grows, we may add more branches, such as an integration branch, where we can test the integration of multiple features before merging them to develop, and run a simpler pipeline (see image below).

Forksïƒ

Forks are used to develop features and bug fixes. They are created from the develop branch by forking to a local repo. When a feature is ready, a PR is created to merge it to develop. When a bug fix is ready, a PR is created to merge it to develop and master.
